一、可用性与可靠性

P0 · 直接影响服务可用

#1 上游请求自动重试 + 指数退避

高影响 中工作量 src/main.py
痛点 当前连接上游失败或返回 5xx 时直接返回错误给客户端。在网络抖动或 DeepSeek 短暂过载时，用户必须手动重试整个对话，体验极差。长时间 reasoning 场景下尤其致命 —— 等了 60 秒后因为一次网络闪断，整轮对话丢失。
方案 在 httpx 调用外层包装重试逻辑：连接错误和 5xx 状态码自动重试（最多 3 次），使用指数退避（1s → 2s → 4s），配合 jitter 防惊群。对 4xx 状态码不重试（API Key 无效等确定性错误）。流式请求在 status_code 校验通过后再开启重试窗口 —— 若流中途断开，重试整个请求。
# 核心实现 @retry( max_attempts=3, wait=exponential_jitter(base=1.0, max=8.0), retry_on=(httpx.ConnectError, httpx.ReadError), status_forcelist={502, 503, 504}, ) async def _call_upstream(payload, api_base, api_key): ...
收益 间歇性故障恢复率从 0% → 95%+，用户无感知自动恢复。
风险 非幂等操作重复执行可能产生副作用。需确认 DeepSeek API 的幂等性保证。

#2 熔断器 (Circuit Breaker)

高影响 中工作量 新增 src/circuit.py
痛点 当上游持续不可用时，每个请求仍然尝试连接，浪费本地资源（端口、内存），并增加上游恢复难度（类似 DDoS）。没有保护机制，雪崩效应会导致代理本身不可用。
方案 实现三态熔断器：Closed → Open → HalfOpen。连续失败 N 次（默认 5 次）进入 Open 状态，直接返回 503 不尝试连接。经过冷却期（默认 30s）进入 HalfOpen，允许少量探测请求通过。探测成功则恢复 Closed，失败则重新 Open。熔断状态变更通过日志告警。
# 状态机 class CircuitBreaker: CLOSED / OPEN / HALF_OPEN failure_threshold = 5 cooldown_seconds = 30 half_open_max = 2
收益 上游故障时保护代理自身资源，故障恢复后自动愈合。
风险 参数调优不当可能导致误熔断或恢复过慢。

#3 请求并发控制与队列

高影响 中工作量 src/main.py
痛点 代理对并发请求数没有任何限制。多个 Codex CLI 实例同时发起流式请求时，可能耗尽上游 API 的并发额度（DeepSeek 通常限制并发数），导致部分请求被拒绝。同时本地 httpx 连接池也存在上限。
方案 引入 asyncio.Semaphore 限制上游并发数（可配置，默认 10）。超出限制的请求排队等待（FIFO），设置等待超时（默认 30s）。返回 `Retry-After` 头告知客户端排队状态。在 /health 端点暴露当前并发数。
收益 避免超出上游并发限制，请求有序排队不丢失。
风险 排队等待时间过长可能导致客户端超时。

#4 API Key 健康度标记与智能路由

中影响 中工作量 src/config.py
痛点 当前 Key 轮询是简单 Round-Robin，对已触发限流或余额不足的 Key 没有感知。一个 Key 被限流后，每轮轮到这个 Key 的请求都会失败，直到人工发现并替换。多 Key 场景下，故障 Key 的存在拉低了整体可用性。
方案 为每个 Key 维护健康状态：连续失败 N 次自动标记为 unhealthy，从轮询池中临时移除，经过冷却期后重新加入。支持通过 /health 端点查看各 Key 状态。可配置 Key 的权重（高频 Key 权重大）。
收益 故障 Key 自动隔离，多 Key 场景可用性显著提升。
风险 需要内存中维护额外状态，重启后丢失（可接受）。

二、性能优化

P1 · 影响响应速度与吞吐

#5 httpx 连接池复用

高影响 低工作量 src/main.py / src/converter/response.py
痛点 当前每次请求都新建 `httpx.AsyncClient()`，非流式路径用完即弃，流式路径在 `stream_generator` 内部也新建。TCP 连接和 TLS 握手无法复用，每个请求额外耗时 50-200ms，高并发时端口耗尽。
方案 在应用启动时创建全局单例 `httpx.AsyncClient`，设置合理的连接池参数（pool_limit=20, keepalive=30s），通过 FastAPI lifespan 管理生命周期。请求之间共享连接池，利用 HTTP/2 或 HTTP/1.1 Keep-Alive 复用连接。
# FastAPI lifespan @asynccontextmanager async def lifespan(app: FastAPI): app.state.http = httpx.AsyncClient( timeout=httpx.Timeout(connect=10, read=120, write=60, pool=10), limits=httpx.Limits(max_connections=20, max_keepalive=10), ) yield await app.state.http.aclose()
收益 非首次请求延迟降低 50-200ms，吞吐提升 3-5 倍。
风险 需要注意连接泄漏和优雅关闭时的连接排空。

#6 reasoning_store 写入优化

中影响 低工作量 src/store.py
痛点 当前每次 `store.append()` 都触发全量 JSON 序列化 + 写盘（`_save()`）。流式请求中每个 assistant turn 都会 append，高频写盘导致 I/O 瓶颈，且随着会话累积、JSON 文件膨胀，`dumps` 耗时线性增长。
方案 (1) 引入写入合并（Debounce）：append 后不立即写盘，而是设置 dirty flag + 定时器（2 秒后批量写入）。(2) 定期清理过期会话（超过 24 小时未访问的 session 自动删除）。(3) 考虑使用 SQLite 替代 JSON 文件，支持增量写入和按需查询。
收益 消除高频写入瓶颈，支持大规模会话管理。
风险 延迟写入可能丢失少量未刷盘数据（进程崩溃时），可接受。

#7 流式响应缓冲与背压控制

中影响 低工作量 src/converter/response.py
痛点 流式生成器中，DeepSeek 产出的每个 token 都立即转换为 SSE 事件 yield 给客户端。如果客户端消费慢（网络不佳或处理慢），上游流会阻塞在 await 上，而生成器的缓冲区无限增长可能导致内存压力。
方案 引入 asyncio.Queue 作为中间缓冲区（maxsize=64），上游解析协程写入队列，下游 SSE 生成协程从队列读取。队列满时上游自动背压（await queue.put 阻塞），防止内存无限增长。
收益 内存可控，慢客户端不影响上游流的处理。
风险 引入额外协程和队列增加代码复杂度。

三、可观测性

P1 · 影响排障效率

#8 Prometheus 指标暴露 + Grafana 看板

高影响 中工作量 新增 src/metrics.py
痛点 当前只有 stdout 结构化日志，没有量化指标。无法回答：QPS 是多少？P50/P90/P99 延迟？流式 vs 非流式占比？各模型调用量分布？上游错误率趋势？Key 级别调用量？这些问题在排障和容量规划时至关重要。
方案 使用 `prometheus_client` 库暴露 `/metrics` 端点，收集以下指标：
• Counter: 请求总数 (按 model/stream/status 分 label)
• Histogram: 请求延迟分布 (区分流式建立时间 vs 首 token 时间 vs 总时间)
• Gauge: 当前并发数、队列长度、熔断器状态
• Counter: reasoning_tokens / prompt_tokens / completion_tokens
收益 可量化观测，支持 Grafana 可视化和 Prometheus 告警。
风险 需引入额外依赖 prometheus_client，高基数 label 需注意。

#9 请求链路追踪 (Request ID / Trace ID)

中影响 低工作量 全局
痛点 当前日志中没有统一的请求标识符。多用户并发时，日志交错输出，无法关联"哪个请求经过了哪些处理步骤"。排查特定用户的问题需要靠时间戳猜测。
方案 每个请求生成唯一的 trace_id (uuid7 或 ulid)，注入到日志 context 中（通过 logging.LoggerAdapter 或 structlog）。trace_id 附加在响应头 `X-Trace-Id` 中返回给客户端。所有日志输出自动带 trace_id 前缀。支持从请求头 `X-Trace-Id` 接收上游 trace_id 实现全链路串联。
收益 端到端请求追踪，日志可检索可关联，排障效率提升 10 倍。
风险 需改造所有日志调用点，logging 配置需调整。

#10 敏感信息脱敏日志

低影响 低工作量 src/logger.py
痛点 `log_conversation` 和 `log_upstream_response` 全量打印用户输入和模型输出，包括可能出现的 API Key、密码、token 等敏感信息。日志可能被采集到集中式平台，存在安全合规风险。
方案 添加可配置的日志级别（debug/info/warn），info 级别对 content 做截断（前 200 字符）+ 正则脱敏（匹配 API Key 格式、密码关键词等）。debug 级别保留全量输出，仅开发环境使用。
收益 满足安全合规要求，日志可安全接入集中式平台。
风险 脱敏规则可能漏掉非标准格式的敏感信息。

四、配置灵活性与安全性

P1 · 影响运维效率与安全合规

#11 环境变量覆盖配置

中影响 低工作量 src/config.py
痛点 配置只能通过 config.yaml 文件加载。在容器化部署（Docker/K8s）场景下，敏感信息（API Key）不应写入文件，而应通过 Secret 环境变量注入。当前无法通过环境变量覆盖 api_keys 等关键配置。
方案 load_config 加载 YAML 后，检查环境变量进行覆盖：`CLI_PROXY_API_KEYS` (逗号分隔)、`CLI_PROXY_API_BASE`、`CLI_PROXY_THINKING_DISABLED` 等。优先级：环境变量 > YAML 文件 > 默认值。
# 环境变量覆盖 api_keys_env = os.getenv("CLI_PROXY_API_KEYS") if api_keys_env: config.api_keys = [k.strip() for k in api_keys_env.split(",") if k.strip()]
收益 兼容 K8s Secret / Docker env，敏感信息不入文件。
风险 环境变量优先级可能导致配置文件被"意外覆盖"。

#12 请求频率限制 (Rate Limiting)

中影响 中工作量 新增 middleware
痛点 没有任何限流机制。单个客户端（有意或无意）可在短时间内发送大量请求，耗尽上游 API 配额，影响其他正常用户。目前没有认证机制，任何人都能访问代理。
方案 基于客户端 IP 或 session_id 实现令牌桶（Token Bucket）限流。默认每客户端 30 req/min（可配置）。返回 429 状态码 + `Retry-After` 头。支持通过 config 设置白名单 IP 不限制。未来可扩展 API Key 认证（通过 `Authorization` 头）。
收益 防止配额滥用，保障多用户公平使用。
风险 IP 限流在 NAT 环境下可能误伤正常用户。

#13 热加载配置（无需重启）

低影响 中工作量 src/config.py
痛点 修改模型映射、添加 API Key、调整 thinking_disabled 都需要重启服务。重启期间所有进行中的请求都会中断，流式对话丢失。对于长期运行的服务，频繁重启是不可接受的。
方案 使用 watchdog 监听 config.yaml 文件变更，变更时重新加载并原子替换 Config 对象（通过 `threading.RLock` 保护）。新增 `POST /admin/reload` 端点手动触发重载。配置文件语法错误时保留旧配置并报警。
收益 配置变更零停机，提升运维效率。
风险 热加载过程中的竞态条件需要仔细处理。

五、功能增强

P2 · 提升使用体验与覆盖面

#14 多上游提供商支持

高影响 高工作量 全局重构
痛点 当前硬编码只支持 DeepSeek API。如果 DeepSeek 服务不可用，或者用户想切换到其他兼容 OpenAI Chat Completions 的提供商（通义千问、Moonshot、本地 vLLM 等），完全无法支持。重度使用者可能希望在不同场景使用不同模型提供商。
方案 抽象 Provider 接口（`BaseProvider`），包含 `chat_completions()` 和 `stream_chat_completions()` 方法。DeepSeek 作为首个实现。通过配置 `providers` 段定义多个上游，模型映射支持 `model → provider:model` 格式。请求时根据模型名动态路由到不同 Provider。
# 配置示例 providers: deepseek: type: openai_compatible api_base: "https://api.deepseek.com" api_keys: [...] qwen: type: openai_compatible api_base: "https://dashscope.aliyuncs.com/compatible-mode/v1" api_keys: [...] model_map: "gpt-5.5": "deepseek:deepseek-v4-pro" "qwen3-max": "qwen:qwen-max"
收益 供应商锁定风险解除，多供应商灵活调度。
风险 不同提供商的微妙差异（字段名、行为）需要适配。

#15 reasoning 内容可选跳过 / 截断

中影响 低工作量 src/config.py + converter
痛点 当前通过 `thinking_disabled` 只能全局关闭/开启思考模式。但实际使用中，简单任务用户想跳过 thinking 节省 token 和延迟，复杂任务又想保留。全局一刀切不够灵活。此外，reasoning_content 可能非常长（数千 token），占据大量带宽。
方案 (1) 支持请求级 thinking 控制：通过 Codex 请求中的 `reasoning` 参数透传到 DeepSeek。(2) 支持 reasoning 截断：配置 `reasoning_max_tokens` 或在 store 中只保留最近 N 轮 reasoning。(3) 支持 reasoning 摘要模式：只保留 reasoning 关键结论。
收益 按需使用 thinking，节省 30-60% token 消耗。
风险 截断可能破坏 reasoning 的连贯性。

#16 响应缓存层

中影响 中工作量 新增 src/cache.py
痛点 开发调试时经常重复发送相同的请求（如 "帮我写一个 sort 函数"）。每次都走完整的上游调用，浪费 API 额度、增加延迟。系统指令通常固定，前几轮对话也常相同。
方案 对非流式请求，计算请求指纹（SHA256(payload)），基于内存 LRU 缓存（TTL=5min, max_size=100）。缓存命中直接返回，跳过上游调用。流式请求不缓存（状态复杂）。支持通过请求头 `X-No-Cache: true` 跳过缓存。
收益 重复请求降低延迟 95%+，节省 API 额度。
风险 缓存 key 需考虑 temperature/tools 等参数差异。

#17 多模态内容透传支持

中影响 中工作量 src/converter/request.py
痛点 当前 `_convert_message` 只处理 `input_text` 类型的 content block。如果用户在 Codex CLI 中发送图片（`input_image`）、文件（`input_file`）等多模态输入，这些 block 会因 `block.get("type") != "input_text"` 被静默丢弃，导致模型看不到图片/文件内容。
方案 扩展 `_convert_message` 支持 OpenAI Vision API 格式的 content 数组。`input_image` → `{"type": "image_url", "image_url": {"url": "data:..."}}`，`input_file` → 提取文本内容注入。multipart content 作为 `content` 数组而非字符串传递。
收益 支持图片输入，覆盖更多使用场景。
风险 Base64 图片可能非常大，需要考虑传输优化。

#18 请求/响应历史审计日志

低影响 低工作量 新增 src/audit.py
痛点 当用户反馈"刚才的回答不对"，运维人员无法回溯当时发了什么请求、上游返回了什么。当前日志是实时流式的，没有持久化的请求-响应对照记录。
方案 可选启用审计日志：将每个请求的完整 payload 和响应以 JSONL 格式写入 `audit/` 目录，按日期分文件（`audit/2026-05-16.jsonl`）。每行一个 JSON 对象包含 timestamp、trace_id、request、response、elapsed_ms。定期清理 N 天前的历史文件。
收益 可回溯历史请求，支撑问题复现和数据分析。
风险 磁盘占用增长，需配合清理策略。

#19 reasoning 可视化 Web 控制台

低影响 高工作量 新增 static/
痛点 reasoning_content 能极大提升对模型行为的理解，但当前完全不可见（只在 debug 日志中出现）。用户想知道模型为什么选择了某个 tool_call 或给出某个回答，无法查看推理过程。
方案 新增一个简单的 Web 页面（挂载在 `/console`），实时展示最近会话的 reasoning_content、文本输出、tool_calls 时间线。通过 SSE 向前端推送实时更新。类似 ChatGPT 的"查看思考过程"功能。
收益 模型行为可解释，调试 prompt 效率大增。
风险 前端开发工作量较大，可能需要引入 WebSocket。

六、运维友好性

P2 · 长期维护与部署体验

#20 优雅关闭 (Graceful Shutdown)

中影响 低工作量 src/main.py
痛点 当前服务收到 SIGTERM/SIGINT 时直接终止，所有进行中的流式请求瞬间断开。用户正在等待的长 reasoning 回答丢失，tool_call 结果没有返回给 Codex CLI。K8s 滚动更新时每次都有请求被强制中断。
方案 监听 SIGTERM/SIGINT 信号，设置 draining 标志。收到信号后：停止接受新请求（返回 503 + `Retry-After`），等待现有请求完成（最长 30s 宽限期），关闭 httpx 连接池，flush store，然后退出。超过宽限期未完成则强制关闭。
收益 滚动更新零中断，用户体验大幅提升。
风险 等待超时需要合理设置，避免无限等待。

#21 就绪探针 + 启动预热

低影响 低工作量 src/main.py
痛点 `/health` 只返回 `{"status": "ok"}`，不检查上游可达性。K8s 探针认为服务已就绪，但实际上可能因为网络或 API Key 问题无法正常代理请求。服务启动后可能立即可用，但连接池还没建立。
方案 新增 `/ready` 端点，检查上游 API 可达性（发送一个轻量模型列表请求），检查 API Key 有效性，检查 store 文件可读写。`/health` 保持轻量只检查进程存活，`/ready` 用于 K8s readinessProbe。
收益 准确的就绪判断，避免流量打到不可用实例。
风险 探针请求本身消耗 API 额度（可缓存结果）。

七、代码架构演进

P3 · 为后续扩展铺路

#22 插件化中间件链

中影响 高工作量 架构级重构
痛点 当前所有逻辑集中在 `proxy_responses` 函数和两个 converter 模块中，扩展一个新功能（如缓存、限流、审计）需要修改核心流程代码。Hooks (before_request / after_response) 散落各处，缺少统一抽象。
方案 定义 Plugin 协议：`pre_request(payload) → payload`、`post_response(response) → response`、`on_error(error)`。支持通过配置启用/禁用插件并排序。内置插件：RateLimiter、Cache、Audit、CircuitBreaker。用户可编写自定义插件放在 `plugins/` 目录。
# Plugin 协议 class BasePlugin: async def on_request(self, payload: dict, ctx: RequestContext) -> dict: ... async def on_response(self, response: dict, ctx: RequestContext) -> dict: ... async def on_error(self, error: Exception, ctx: RequestContext) -> None: ...
收益 开闭原则，新功能通过插件添加，核心代码稳定。
风险 过度设计风险，当前规模可能不需要如此复杂的抽象。
